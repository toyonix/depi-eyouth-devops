## Theory Questions

### a) Prometheus Fundamentals

#### What are metrics in Prometheus, and how are they collected?

The metrics are data gathered about a certain service or system. It is used to monitor how the system performs during its life time. Stuff like cpu and ram usage, network activity, and up time.

It is collected using Prometheus integrations from the dashboard or even you can create your own custom integration inside of it if you want or have to.

#### Explain the difference between counter, gauge, histogram, and summary metric types in Prometheus. Give examples of each.

All of these are metric types that you use inside of Prometheus when collecting data. 

Counter: From its name, it is a cumulative metric that represents a monotonically increasing value. It can only be incremented and is reset to zero on restart. It is commonly used for things like counting errors or requests.

Gauge: It represents a single numerical value that can go up or down. It reflects the current state of something, like CPU usage or memory usage.

Histogram: It represents the distribution of values by grouping observations into buckets (ranges). It also provides the total count and sum of all observed values. It is commonly used for things like request durations or response sizes.

Summary: It is similar to a histogram but calculates quantiles (e.g., p50, p95, p99) directly. It also provides the total count and sum of observations, giving a statistical summary of the data.

### b) Prometheus in Kubernetes

#### How can Prometheus monitor Kubernetes applications? Explain how Prometheus scrapes metrics from pods and nodes in the cluster.

Prometheus monitor Kubernetes applications using metrics server using the Kubernetes API. In that api there is `/metrics` end point that we use to get all metrics about a pod. So, Prometheus sends an http request on this endpoint and the response is the metrics that Prometheus stores.

#### What is Alertmanager, and how is it used alongside Prometheus?

Alertmanager is a separate service that works alongside Prometheus. Its role is to handle alerts generated by Prometheus and notify the appropriate stakeholders when certain conditions are met. Prometheus evaluates alerting rules based on collected metrics, and when a condition indicates a problem (e.g., high error rate or CPU usage), it sends an alert to Alertmanager. Then the Alertmanager sends an email or message on any configured platform about the situation.

### c) Monitoring Strategies

#### Describe a monitoring strategy for a production-grade application. What are key metrics you would monitor, and how would you set up alerting for critical failures?

A production-grade monitoring strategy focuses on tracking key metrics like latency, traffic, error rates, and resource utilization to ensure system health and performance.

Metrics are collected using Prometheus from application endpoints and infrastructure components, and alerting rules are defined to detect abnormal conditions such as high error rates, increased latency, or system downtime. When these conditions are met, Prometheus sends alerts to Alertmanager, which handles grouping, deduplication, and routing notifications to the appropriate stakeholders, ensuring timely response to critical failures while minimizing noise.
